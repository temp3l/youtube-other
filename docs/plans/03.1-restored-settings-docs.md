Audit and update the restored story-production documentation and prompt templates so they accurately reflect the repository after implementation of Tasks 01–04.

Files to review:

- `docs/templates/audio/system-prompt.md`
- `docs/templates/audio/short-story-prompt.md`
- `docs/multilingual-story-localization-settings.md`
- `docs/cli.md`

Context:

- These files were missing from the checkout and have now been restored.
- They may be outdated.
- Task 04 introduced genre policies, full-story contracts, deterministic policy/version fingerprints, and stable serializer/hash semantics.
- Earlier tasks introduced or refined StoryIR, source cleaning, provenance handling, localization flow, and artifact contracts.
- Do not assume the restored files are authoritative.
- The current implementation, tests, schemas, CLI registration, configuration layer, and persisted artifact contracts are the source of truth.

Objectives:

1. Determine whether each restored file is current, partially stale, or obsolete.
2. Update the files so they accurately describe and support the implemented repository behavior.
3. Preserve backward-compatible public behavior unless a documented bug is proven.
4. Add or update tests where these files are loaded or validated at runtime.
5. Do not implement later roadmap tasks.

Required analysis

For each restored file:

1. Trace all repository references:

   - imports and filesystem reads;
   - CLI commands;
   - package scripts;
   - tests and fixtures;
   - prompt loaders;
   - localization services;
   - audio generation;
   - story and short-story generation;
   - configuration defaults;
   - documentation links.

2. Compare the file contents against:

   - current TypeScript types and schemas;
   - StoryIR fields and invariants;
   - full-story contract fields;
   - genre-policy registry behavior;
   - localization language support and canonical flow;
   - prompt compiler inputs;
   - metadata and audio stage separation;
   - model configuration from `.env` and config modules;
   - current CLI command names and options;
   - current artifact paths and resume behavior.

3. Identify:

   - stale command names;
   - invalid paths;
   - removed or renamed fields;
   - undocumented required fields;
   - obsolete prompt instructions;
   - duplicated responsibilities;
   - instructions that conflict with the current contract;
   - language lists that no longer match implementation;
   - model names or reasoning settings that should not be hard-coded;
   - metadata or audio instructions incorrectly included in story prompts;
   - assumptions that bypass StoryIR or the prompt compiler;
   - claims that are not enforced by code;
   - documentation that describes planned rather than implemented behavior.

File-specific expectations

## `docs/templates/audio/system-prompt.md`

Verify that it:

- contains only audio/TTS-stage responsibilities;
- does not instruct story rewriting, localization, metadata generation, scene generation, or image generation;
- uses the current audio input contract;
- does not require fields unavailable at the audio stage;
- does not hard-code a provider or model unless the runtime truly requires it;
- clearly distinguishes full-story and short-form narration where applicable;
- preserves text exactly when the audio stage is expected to narrate finalized content;
- does not mutate wording, translate text, add commentary, or produce metadata.

## `docs/templates/audio/short-story-prompt.md`

Verify that it:

- is actually consumed by the current short-story or short-audio flow;
- does not duplicate the main story rewrite prompt;
- accepts the current finalized short-story artifact or contract;
- contains no metadata, image, scene, or localization responsibilities;
- has clear output constraints;
- does not ask the audio model to repair, summarize, or rewrite content unless this is explicitly implemented;
- matches the current supported-language behavior.

If the file name or responsibility is misleading, recommend a rename, but do not rename it unless all references, tests, and documentation can be updated safely in the same change.

## `docs/multilingual-story-localization-settings.md`

Verify that it accurately documents:

- supported languages;
- canonical source language;
- canonical localization flow;
- whether English is copied, rewritten, or normalized;
- whether Spanish, German, Portuguese, and any other languages are generated from canonical English or directly from the original source;
- separation between full-story localization and short-story localization;
- retry and repair behavior;
- maximum-output-token handling;
- completeness validation;
- locale codes and filename conventions;
- artifact locations;
- resume behavior;
- configured model selection;
- configuration precedence;
- whether metadata and audio are excluded from localization prompts;
- provenance and source-language preservation;
- contract validation before persistence.

Do not add language support that is not implemented.

Implementation rules

- Prefer small, evidence-based edits.
- Do not rewrite all documentation stylistically unless required for correctness.
- Preserve useful historical explanations where still accurate.
- Mark deprecated behavior explicitly instead of silently presenting it as current.
- Do not create placeholder content.
- Do not hard-code current OpenAI model names in documentation when the repository supports configuration.
- When showing model examples, label them as examples and reference the configuration keys that control them.
- Do not add undocumented environment variables.
- Do not change runtime behavior merely to make restored documentation appear correct.
- If code and documentation disagree, determine which one reflects the approved Tasks 01–04 contracts before changing either.
- Any runtime change must be minimal, justified, tested, and directly related to reconciling these restored files.
